üíæ Konsep Dasar Transaksi dan ACID Properties

Baik, saya akan mulai dengan menjelaskan konsep dasar transaksi dalam basis data.
Transaksi adalah sebuah rangkaian operasi basis data yang dilakukan sebagai satu kesatuan logis. Transaksi memiliki karakteristik penting yang disebut dengan ACID Properties. Mari kita jelaskan satu persatu:

üîê ACID Properties

A - Atomicity (Atomisitas): Sebuah transaksi harus dilaksanakan seluruhnya atau tidak sama sekali. Tidak ada kondisi transaksi yang sebagian selesai. Jadi, jika ada kegagalan di tengah jalan, seluruh transaksi akan dibatalkan dan database akan kembali ke keadaan semula. Konsep ini seperti "semua atau tidak sama sekali" (all or nothing).
C - Consistency (Konsistensi): Data di dalam basis data harus tetap bersifat konsisten. Artinya, transaksi harus membawa database dari satu state yang konsisten ke state konsisten lainnya. Semua aturan integritas database harus terpenuhi sebelum dan sesudah transaksi dijalankan. Tidak boleh ada inkonsistensi data.
I - Isolation (Isolasi): Setiap transaksi harus terisolasi dan tidak boleh diganggu oleh transaksi lain. Artinya, saat satu transaksi sedang berjalan, transaksi lain tidak boleh melihat atau mempengaruhi data yang sedang diproses. Setiap transaksi harus berjalan independen tanpa interferensi dari transaksi lain.
D - Durability (Daya Tahan/Persistensi): Setelah transaksi selesai dan commit, data harus tetap persisten dan tahan terhadap kegagalan sistem. Data yang telah di-commit harus tersimpan dengan aman di disk dan tidak akan hilang meskipun terjadi kegagalan atau crash pada sistem.

üîÑ State Transaksi

Dalam sebuah transaksi, terdapat 5 state yang mungkin terjadi:
No State Keterangan
1 Active Transaksi sedang berjalan dan melaksanakan operasi
2 Partially Committed Transaksi telah menyelesaikan eksekusi tetapi belum disimpan ke database
3 Committed Transaksi berhasil selesai dan perubahan telah disimpan ke database
4 Failed Transaksi mengalami kegagalan selama eksekusi
5 Aborted Transaksi dibatalkan dan database dikembalikan ke state sebelum transaksi dimulai
üí° Penting: Dalam praktikum ini, kami akan menggunakan basis data MySQL dengan storage engine InnoDB yang mendukung transaksi ACID penuh.

üõ†Ô∏è Perintah Transaksi dalam MySQL

MySQL menyediakan beberapa perintah untuk mengelola transaksi:
Perintah Fungsi
BEGIN / START TRANSACTION Memulai sebuah transaksi baru
COMMIT Menyimpan dan mengakhiri transaksi
ROLLBACK Membatalkan transaksi dan mengembalikan data ke state sebelumnya
SAVEPOINT Membuat titik penyimpanan dalam transaksi
‚ö†Ô∏è Catatan Penting: Pastikan tabel-tabel yang digunakan menggunakan storage engine InnoDB. Jika menggunakan MyISAM, transaksi tidak akan bekerja. Untuk mengecek: SELECT ENGINE FROM INFORMATION_SCHEMA.TABLES WHERE TABLE_NAME='nama_tabel';

üìù Penjelasan Commit

COMMIT adalah perintah yang menandakan bahwa transaksi telah selesai dilaksanakan dan semua perubahan data menjadi permanen di database. Artinya, setelah COMMIT dijalankan, data tidak bisa lagi dibatalkan dengan ROLLBACK.
Karakteristik COMMIT:
‚Ä¢ Membuat perubahan data menjadi permanen
‚Ä¢ Melepaskan lock pada data yang digunakan
‚Ä¢ Menyelesaikan transaksi
‚Ä¢ Data yang di-commit akan terlihat oleh transaksi lain

‚Ü©Ô∏è Praktikum 2: Perintah ROLLBACK

Sekarang kita akan mempelajari perintah ROLLBACK yang berguna untuk membatalkan perubahan data.

üìù Penjelasan Rollback

ROLLBACK adalah perintah yang digunakan untuk membatalkan transaksi yang sedang berjalan dan mengembalikan database ke state sebelum transaksi dimulai. Pernyataan ROLLBACK hanya berpengaruh jika transaksi belum di-COMMIT. Setelah COMMIT, ROLLBACK tidak bisa lagi membatalkan perubahan.
Karakteristik ROLLBACK:
‚Ä¢ Membatalkan semua perubahan dalam transaksi yang belum di-COMMIT
‚Ä¢ Mengembalikan database ke state awal sebelum transaksi
‚Ä¢ Melepaskan lock pada data yang digunakan
‚Ä¢ Hanya bekerja untuk transaksi yang belum di-COMMIT

üîñ Praktikum 3: SAVEPOINT (Checkpoint)

Sekarang kita akan mempelajari SAVEPOINT, yang memungkinkan kita untuk membuat titik penyimpanan dalam transaksi.

üìù Penjelasan Savepoint

SAVEPOINT (juga dikenal sebagai Checkpoint) adalah titik dalam transaksi di mana database state disimpan. Berbeda dengan ROLLBACK yang membatalkan seluruh transaksi, ROLLBACK TO SAVEPOINT hanya membatalkan kembali ke savepoint yang ditentukan, bukan ke awal transaksi.
Karakteristik SAVEPOINT:
‚Ä¢ Membuat titik penyimpanan dalam transaksi
‚Ä¢ Memungkinkan ROLLBACK parsial ke savepoint tertentu
‚Ä¢ Tidak memperkenalkan COMMIT otomatis
‚Ä¢ Berguna untuk transaksi kompleks dengan banyak operasi

üîß Langkah-Langkah Praktik

1 Buat Transaksi 5 dengan SAVEPOINT
Membuat transaksi dengan tiga INSERT dan menggunakan SAVEPOINT
Saya akan membuat transaksi 5 yang melakukan 3 kali INSERT data pasien: 1. INSERT pasien id 7 2. Buat SAVEPOINT point1 3. INSERT pasien id 8 4. ROLLBACK ke point1 (membatalkan INSERT id 8) 5. INSERT pasien id 9 6. COMMIT (menyimpan id 7 dan id 9, tetapi id 8 dibatalkan)

  INSERT INTO pasien
  (id_pasien, nama_pasien, alamat_pasien, jenis_kelamin)

  ROLLBACK TO point1;

  INSERT INTO pasien
  (id_pasien, nama_pasien, alamat_pasien, jenis_kelamin)
  VALUES (9, 'Siti', 'Jl Pemuda', 2);

  COMMIT;
END

‚ö° Praktikum 4: TRIGGER

Sekarang kita akan mempelajari salah satu fitur paling powerful dalam SQL: TRIGGER.

üìù Penjelasan Trigger

TRIGGER adalah sebuah event atau pemicu yang secara otomatis menjalankan perintah SQL ketika terjadi perubahan data pada tabel tertentu. TRIGGER dapat dipicu oleh operasi INSERT, UPDATE, atau DELETE. Selain itu, TRIGGER juga dapat dikonfigurasi untuk dijalankan sebelum (BEFORE) atau sesudah (AFTER) operasi tersebut.
Karakteristik TRIGGER:
‚Ä¢ Dijalankan otomatis ketika terjadi event tertentu
‚Ä¢ Event: INSERT, UPDATE, DELETE
‚Ä¢ Timing: BEFORE atau AFTER event
‚Ä¢ Berguna untuk memelihara integritas data dan otomasi
‚Ä¢ Dapat melakukan validasi dan penyesuaian data

üìã Jenis-Jenis Trigger

No Jenis Trigger Kapan Dijalankan
1 BEFORE INSERT Sebelum data dimasukkan ke tabel
2 AFTER INSERT Setelah data dimasukkan ke tabel
3 BEFORE UPDATE Sebelum data diubah
4 AFTER UPDATE Setelah data diubah
5 BEFORE DELETE Sebelum data dihapus
6 AFTER DELETE Setelah data dihapus

3 Buat Trigger AFTER INSERT pada Tabel transaksi_obat
Membuat trigger yang otomatis mengurangi stok obat
Sekarang saya akan membuat trigger yang sangat berguna. Trigger ini akan otomatis mengurangi stok obat ketika ada transaksi pembelian. Scenario-nya adalah: ketika seorang pasien membeli obat Amoxillin sebanyak 5 buah, maka secara otomatis stok Amoxillin akan berkurang dari 20 menjadi 15.
DELIMITER $$
CREATE TRIGGER tg_jual AFTER INSERT
ON transaksi_obat
FOR EACH ROW
BEGIN
  UPDATE obat
  SET stok = stok - new.jumlah
  WHERE id_obat = new.id_obat;
END $$
DELIMITER ;
Mari kita bahas struktur trigger ini: ‚Ä¢ DELIMITER $$ - Mengubah delimiter dari ; menjadi $$ ‚Ä¢ CREATE TRIGGER tg_jual - Membuat trigger dengan nama tg_jual ‚Ä¢ AFTER INSERT - Trigger dijalankan setelah INSERT ‚Ä¢ ON transaksi_obat - Trigger bekerja pada tabel transaksi_obat ‚Ä¢ FOR EACH ROW - Untuk setiap baris yang diinsert ‚Ä¢ BEGIN...END - Blok pernyataan SQL ‚Ä¢ UPDATE obat SET stok = stok - new.jumlah - Mengurangi stok obat ‚Ä¢ WHERE id_obat = new.id_obat - Dengan id obat yang sesuai dengan data baru

üîÑ Praktikum 5: Basis Data Terdistribusi - Replikasi Master-Slave

Sekarang kita akan memasuki bagian kedua praktikum: Basis Data Terdistribusi dengan replikasi Master-Slave.

üìù Penjelasan Basis Data Terdistribusi

Basis Data Terdistribusi adalah sistem database yang tersebar di beberapa lokasi fisik dan terhubung melalui jaringan. Dalam konteks praktikum ini, kita akan melakukan replikasi data dari satu server (Master) ke server lain (Slave). Artinya, semua perubahan data pada Master akan otomatis terefleksi pada Slave.
Keuntungan Replikasi Database:
‚Ä¢ Redundansi: Data tersimpan di multiple lokasi
‚Ä¢ Availability: Sistem tetap berjalan jika salah satu server down
‚Ä¢ Load Balancing: Distribusi beban kerja ke multiple server
‚Ä¢ Disaster Recovery: Backup otomatis di lokasi berbeda
‚Ä¢ Scalability: Dapat menangani lebih banyak read requests

üèóÔ∏è Arsitektur Master-Slave

Dalam praktikum ini, kita akan menggunakan arsitektur Master-Slave dengan: ‚Ä¢ Master: Windows 10 (database utama, menerima write operations) ‚Ä¢ Slave: Ubuntu (database replika, menerima perubahan dari Master) ‚Ä¢ Data mengalir satu arah: Master ‚Üí Slave

‚ö†Ô∏è Jika Ada Status "No": Jika ada salah satu status yang "No", berarti ada masalah koneksi. Periksa: ‚Ä¢ IP address master ‚Ä¢ Username dan password replikasi ‚Ä¢ Firewall/network connectivity ‚Ä¢ Binary log file dan position
